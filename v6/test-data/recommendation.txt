Issues found in provider_deepseek.py
======================================

🔴 Potential Issues
-------------------

1. Missing error handling for missing keys (crash risk)
   In from_deepseek_tool_call and from_deepseek_assistant_message, dictionary keys
   are accessed directly (e.g., tool_call_data["function"], choice["message"]) instead
   of using .get(). If the API response format changes or is missing an expected field,
   the code will throw a KeyError and crash.

2. No handling for None content in to_deepseek_message
   For system and user messages, message.content is passed directly. If content is None,
   DeepSeek's API may reject the request (it often expects a string, even if empty).

3. to_deepseek_message — tool role assumes tool_call_id exists
   The tool message branch references message.tool_call_id without checking if it's
   None. If it's missing, the API could reject it.

4. Dead / commented-out code
   There's an old commented-out version of from_deepseek_assistant_message that's no
   longer used. It clutters the file and could confuse future developers.
   Also, the comment inside it is misleading — DeepSeek uses reasoning_content in
   addition to content, not instead of it.

5. from_deepseek_assistant_message — content could be None but passed as-is
   deepseek_msg.get("content") could return None. If AssistantMessage expects a string,
   this could cause issues downstream.

6. No logging or validation
   There's no logging, input validation, or graceful fallbacks anywhere. If something
   goes wrong (e.g., malformed API response, missing fields), the error message will be
   a raw Python exception rather than something informative.

🟡 Minor / Cosmetic Issues
--------------------------

- No docstrings — the functions lack documentation, making it harder to understand the
  expected input/output formats.
- Inconsistent key access — from_deepseek_assistant_message mixes direct access
  (choice["message"]) with .get() (choice.get("finish_reason")) — inconsistent style.

✅ What's Good
--------------

- The separation of concerns (to/from conversion) is clean.
- The list comprehension for converting multiple messages/tool calls is concise.
- The copy (deepseek_msg = choice["message"].copy()) is good practice to avoid mutating
  the original.

💡 Suggested Improvements
--------------------------

1. Use .get() with safe defaults when parsing API responses.
2. Remove the dead commented-out code.
3. Add a fallback for content (e.g., empty string "").
4. Add basic input validation or logging.
